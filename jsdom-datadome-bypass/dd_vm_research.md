# DataDome VM (plv3) Research - Key Findings

## From glizzykingdreko's analysis (Jan 2026):

### The plv3 parameter
- Generated by DataDome's `vm-obj.js` file
- Uses a full JavaScript Virtual Machine (VM) with ~100 opcodes
- Executes ~360k instructions for captcha, ~260k for interstitial
- Output: `{"r": "fingerprint_value", "i": instruction_count, "t": elapsed_time}`
- `r` = plv3 value, `i` and `t` used in signals payload

### Key insight about solving:
> "Currently the bytecode instructions makes several checks before/during the fingerprint extraction, 
> that not only needs to match the details of your solved payload, but also of the challenge you are solving."

> "This means that using a browser tab for solving it isn't just the worst and unscalable solution, 
> will be also non-working."

### The recommended approach:
> "The actual best approach would be to understand those instructions, what the VM is looking for 
> and analysing, in order to then create a script-based env able to correctly run the file with 
> the same details used for the signals payload constructions."

### What this means for us:
1. The VM checks are tied to the specific challenge parameters
2. A browser tab alone won't work - need to understand the VM
3. Need a "script-based env" that matches the signals payload
4. TakionAPI is a commercial service that solves this

### Commercial solutions:
- TakionAPI - specializes in DataDome bypass
- CapSolver - has DataDome support
- These services handle the VM reverse engineering

## Critical Question:
Our jsdom approach was on the right track! The VM runs JS code that collects fingerprints.
If we can make jsdom's environment match what the VM expects, it should work.

The issue is: the VM bytecode checks are dynamic and tied to challenge params.
We need to either:
1. Reverse engineer the VM bytecode (very complex, changes often)
2. Use a real JS engine with proper DOM mocks (our jsdom approach, but better mocks)
3. Use a commercial API service (TakionAPI, CapSolver)
