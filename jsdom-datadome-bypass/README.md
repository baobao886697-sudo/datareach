## 1. Executive Summary

This document outlines a promising, lightweight, and browser-less approach to bypass the Cloudflare and DataDome protections on TruePeopleSearch (TPS). The core strategy is to use `curl_cffi` for efficient HTTP requests and a `jsdom` environment within Node.js to solve DataDome's JavaScript (JS) challenges.

This method avoids the high memory overhead of full browser instances (e.g., Camoufox), enabling significantly higher concurrency on smaller servers. While the end-to-end flow is nearly complete, the final step is blocked by DataDome returning a `captcha` view instead of a `redirect`, indicating that the fingerprint generated by `jsdom` is not yet trusted. This document details the progress, the current blocker, and the specific next steps required to achieve a fully functional bypass.

## 2. Architecture Overview

The proposed architecture is a multi-step, purely script-based flow that requires no browser processes:

| Step | Component | Action | Purpose |
| :--- | :--- | :--- | :--- |
| **1** | `curl_cffi` (Python) | `GET` TPS search URL | Trigger Cloudflare/DataDome protection. |
| **2** | `curl_cffi` (Python) | Parse 403 response | Extract DataDome challenge parameters (`dd` object). |
| **3** | `curl_cffi` (Python) | `GET` interstitial URL | Fetch the 434KB JS challenge page from `geo.captcha-delivery.com`. |
| **4** | `Node.js` + `jsdom` | Execute the challenge JS | Run the 434KB JS in a mocked browser environment to generate the solution payload. |
| **5** | `curl_cffi` (Python) | `POST` payload | Submit the solved challenge to DataDome's interstitial endpoint. |
| **6** | `curl_cffi` (Python) | Receive `datadome` cookie | Get the valid session cookie from DataDome. |
| **7** | `curl_cffi` (Python) | `GET` TPS search URL | Re-request the original page with the valid cookie to get data. |

This architecture is designed to be highly scalable, with each "worker" being a lightweight Python coroutine that orchestrates these steps.

## 3. Progress and Current Status

We have successfully implemented and validated **90%** of this architecture. The following steps are fully functional:

- **Step 1: Cloudflare Bypass & Challenge Retrieval:** Using `curl_cffi` with the `chrome142` impersonation profile successfully bypasses Cloudflare's initial TLS fingerprinting and retrieves the 403 response containing the DataDome interstitial challenge.

- **Step 2: Interstitial Page Fetch:** The `dd` object from the 403 response is correctly parsed to construct and fetch the full 434KB interstitial challenge page.

- **Step 3: `jsdom` Challenge Execution:** The core of the solution works. The 434KB of obfuscated JavaScript successfully runs within a `jsdom` environment. We have mocked essential browser APIs (`navigator`, `screen`, `window`, etc.) to a degree that allows the script to execute without crashing.

- **Step 4: Payload Generation & Submission:** The `jsdom` environment correctly intercepts the `XMLHttpRequest` made by the challenge script, capturing a **~6.8KB payload**. This payload is then successfully `POST`ed to DataDome's servers.

- **Step 5: DataDome Acknowledgment:** DataDome's servers respond with a `200 OK` to our `POST` request, indicating that the payload format is valid and has been accepted for processing.

### The Current Blocker: `"view": "captcha"`

The final and only blocking issue is the content of DataDome's `200 OK` response. Instead of providing a valid cookie and a `redirect` view, it returns:

```json
{"view": "captcha", "url": "..."}
```

This means that while DataDome accepts our payload, its trust scoring system has flagged our generated fingerprint as suspicious or incomplete, thereby requiring a full CAPTCHA solve. The root cause is that our mocked browser environment in `jsdom` is not yet realistic enough.

## 4. DataDome Detection Mechanism Deep-Dive

Understanding what DataDome checks is essential for improving the `jsdom` mock. Based on our research, DataDome's trust scoring system evaluates the following components:

### 4.1 TLS Fingerprinting (JA3/JA4)

DataDome inspects the TLS handshake to determine the client's identity. Different browsers and HTTP libraries produce unique TLS fingerprints. Using `curl_cffi` with the `chrome142` impersonation profile successfully passes this check. **This is already solved.**

### 4.2 HTTP/2 Fingerprinting

Beyond TLS, DataDome also inspects HTTP/2 frame priorities (SETTINGS, WINDOW_UPDATE, PRIORITY frames). The `curl_cffi` library handles this correctly with its impersonation profiles. **This is already solved.**

### 4.3 JavaScript Fingerprinting (The Blocker)

This is the core of DataDome's detection and the reason our `jsdom` approach currently fails. The interstitial challenge JS collects:

| Fingerprint Type | What It Checks | Our Mock Status |
| :--- | :--- | :--- |
| **Canvas** | Draws shapes/text, hashes the pixel data (`cfpfe`) | Static mock (**needs real canvas**) |
| **WebGL** | GPU vendor/renderer, shader compilation results | Basic mock (needs expansion) |
| **AudioContext** | Oscillator/analyser node output hash | Minimal mock (needs improvement) |
| **Navigator** | `userAgent`, `platform`, `hardwareConcurrency`, `deviceMemory`, `languages` | Mocked (good) |
| **Screen** | `width`, `height`, `colorDepth`, `pixelDepth` | Mocked (good) |
| **Timezone** | `Intl.DateTimeFormat().resolvedOptions().timeZone` | Default (may need setting) |
| **Fonts** | Available font list detection | Not mocked |
| **Storage** | `localStorage`, `sessionStorage`, `indexedDB` availability | Provided by jsdom |
| **Plugins** | `navigator.plugins` list | Not mocked |

### 4.4 The `plv2` and `plv3` Parameters

The interstitial challenge generates two critical parameters:

- **`plv2`**: Contains the browser fingerprint data (Canvas hash, WebGL info, etc.). This is where our mock quality matters most.
- **`plv3`**: Contains the proof-of-work result. This is generated by DataDome's VM (Virtual Machine) which executes ~260K instructions for interstitial challenges. The VM checks are tied to the specific challenge parameters and cross-reference with the `plv2` fingerprint.

### 4.5 The VM Obfuscation Layer

As of Feb 2026, DataDome has deployed VM-based obfuscation for their Device Check and Slider challenges [1]. The VM:

- Uses ~100 custom opcodes
- Executes 260K+ instructions for interstitial challenges
- Cross-validates fingerprint data between `plv2` and `plv3`
- Changes signal names daily (dynamic key rotation)

This means the `jsdom` environment must be consistent: the fingerprint data collected during `plv2` generation must match what the VM observes during `plv3` generation.

## 5. Next Steps: Achieving a Trusted Fingerprint

The immediate goal is to improve the fidelity of the `jsdom` environment to generate a trusted fingerprint. Based on our research, the key areas to focus on are:

### 5.1 Implement Real Canvas Fingerprinting (Highest Priority)

**Problem:** Our current `canvas` mock is static. DataDome's JS performs drawing operations and hashes the resulting image data to generate a unique fingerprint (`cfpfe`).

**Solution:** Integrate a real Canvas API implementation for Node.js:

```bash
# Option 1: node-canvas (requires native build)
sudo apt-get install -y build-essential libcairo2-dev libjpeg-dev libpango1.0-dev libgif-dev librsvg2-dev
pnpm add canvas

# Option 2: @napi-rs/canvas (prebuilt binaries, easier)
pnpm add @napi-rs/canvas
```

Then replace the mock canvas in `beforeParse`:

```javascript
const { createCanvas } = require('@napi-rs/canvas');

// In beforeParse(window):
const origCreateElement = window.document.createElement.bind(window.document);
window.document.createElement = function(tag) {
    const el = origCreateElement(tag);
    if (tag.toLowerCase() === 'canvas') {
        const realCanvas = createCanvas(el.width || 300, el.height || 150);
        el.getContext = function(type) {
            if (type === '2d') return realCanvas.getContext('2d');
            // WebGL still needs mock
            return null;
        };
        el.toDataURL = () => realCanvas.toDataURL();
    }
    return el;
};
```
